# Research Notes: 跨设备图片导出文字变大与换行兼容性原因分析

## 1. 深度技术原因剖析

### 1.1 核心机制：DPR (Device Pixel Ratio) 缩放与浏览器“最小字号限制”冲突
目前我们在导出长图时，为了避免输入框和选择框在图片中显得丑陋，采用了 **“动态替换 DOM”** 的方案：
1. 用一个普通的 `<div>` 去替换页面上的 `<input>` 或 `<select>`。
2. 为了保证这个新 `<div>` 的外观与原表单元素完全一致，代码使用 `window.getComputedStyle(element)` 读取了原表单元素的计算样式（包括 `fontSize`, `padding`, `width`, `height` 等）。
3. 由于在高缩放（DPR > 1，例如 125%、150% 等）的屏幕上，浏览器读取到的 `getComputedStyle` 的 px 都是已经被系统放大的物理像素，因此代码使用 `adjustPx` 辅助函数将所有 px 值除以 `devicePixelRatio` 进行了缩小。
4. **冲突爆发点**：如果用户屏幕的缩放比例是 150%（`dpr = 1.5`），页面上原本 `12px` 的文字，计算样式中读取到的是 `18px`，除以 1.5 还原为 `12px`；如果是 `10px` 的文字，除以 1.5 变成 `6.6px`。甚至有时代码逻辑试图将其缩小到更小的值（如 8px、10px）。
5. 此时，Chrome 浏览器（包括 Electron 的 Chromium 内核）会强制执行**“最小字号保护限制”**（一般中文字体环境下强制限制最小为 `12px`）。因此，虽然样式声明了 `fontSize: 8px`，但浏览器渲染出来的字号实际依然是 `12px`。
6. **结果**：容器的宽度 and 高度按除法切实缩小了 33%，而字体的物理尺寸却被拦截在 `12px` 拒绝缩小。这相当于字号变相“变大”了 1.5 倍，从而直接导致了文本容器因容纳不下文字而把字体挤压换行，甚至卡片错落、对不齐。

### 1.2 系统默认字体的字符宽度差异 (Font Metrics)
- 在没有安装“微软雅黑”或特定无衬线中文字体的部分老旧电脑上，浏览器会退回到宋体或其他系统兜底字体。
- 退回字体中，个别中文字符及标点符号的字宽（Glyph Width）天生比微软雅黑宽，例如 Badge 里的“有极限蛋”这四个字，在某些字体下会多占几个像素的宽度，从而在原本已经缩水且没有富余空间的容器中发生了折行。

### 1.3 非整数像素渲染四舍五入
- 当 `window.devicePixelRatio` 为分数（例如 `1.25` 或 `1.5`）时，所有像素转换（例如 `15px / 1.5 = 10px`，而某些 `25px / 1.5 = 16.666px`）在浏览器进行 SVG 转换与 canvas 渲染时都会进行四舍五入取整。
- 这会导致元素出现 1 像素以内的偏移或抖动，在长图这种多行、多卡片的结构中累加起来就会导致右侧对不齐。

### 1.4 html2canvas 中的垂直对齐与动画问题
- **文字上移问题**：在 html2canvas 渲染过程中，子元素的内联或 padding 布局计算与主浏览器稍有不同，如果容器使用简单的 `py-1` 结合 `flex items-center`，基线解析容易失准，引发文本上移。使用固定高度（如 `h-7`）、显式 flex 盒模型并配合 `leading-none`（确保行高不参与干扰）能使 html2canvas 的基线定位更稳健。
- **图标上移与动画问题**：`animate-bounce` 带有 `transform: translateY(...)` 的帧动画，在截图导出时，由于截图瞬间元素处于动画轨迹的不同帧，容易被捕获到高处；此外，复杂的 transform 变换和 keyframe 动画在 html2canvas 的节点树克隆中极易被错误还原或渲染偏移。因此在导出场景中，应彻底移除这类图标动画并设置其为 `shrink-0`。

## 2. 解决方案方向

### 方案 A：继承 Tailwind CSS 类名（逻辑自适应，最推荐）
- **实现**：替换 DOM 时，不再读取 px 样式并除以 DPR，而是让新创建的静态 `<div>` 直接继承原组件身上所有的 Tailwind 类名（如 `w-full`, `text-xs`, `px-2` 等）。
- **原理**：让布局保持在 1200px 宽度内自适应渲染，不再人工干预具体 px 大小。因为不涉及物理像素除法，字号稳定在标准的 `text-xs` (12px)，彻底避免了触发最小字号限制，完美解决折行问题，且保留了原本极其优秀的自适应版面。

### 方案 B：Electron 后台隐藏窗口 Offscreen 100% 渲染与截图（桌面端绝对完美）
- **实现**：主进程起一个固定 `--force-device-scale-factor=1` 的后台隐藏窗口。
- **原理**：直接将宠物数据传给隐藏窗口，隐藏窗口由于被强制 100% 缩放，永远不会发生 DPR 缩放造成的偏差，接着通过 Electron 原生的 `capturePage()` 截取，图片绝对完美。

### 3. 后续优化：导出格式与兼容性
- **图片格式优化**：当前项目普遍使用 `PNG` 格式导出，在高分辨率下（如 2 倍或 3 倍的 Canvas）会导致文件体积过大（甚至超过 5MB），在分享给微信等 IM 时会遭遇自动压缩。后续应引入 `HTMLCanvasElement.toBlob('image/jpeg', 0.9)`，通过 JPG 格式显著降低文件体积，同时避免透明通道带来的边缘白边残留问题。
- **对齐稳定性增强**：针对右侧对齐问题，除了移除动画外，应统一使用 `box-border` 盒模型，并确保容器宽度为 `4px` 的倍数，以配合浏览器渲染引擎的渲染步长，进一步降低 sub-pixel 渲染带来的累积误差。
